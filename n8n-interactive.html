💡 核心洞见

在「逐条处理」模式下，每个节点对每条数据独立执行一次。3 条数据 = HTTP请求执行 3 次 = 3 次独立的 API 调用。

这就是 n8n 的本质：节点不是处理"一批数据"，而是对每一条数据分别执行。表达式中的 $json 永远指向当前正在处理的那一条。

⚠ 常见误区：以为一个节点对整个数组只执行一次。实际上默认模式下每条数据各执行一次。

    • 分支与合并 Lv4 迁移应用

    真实场景需要条件分支。IF 节点将数据路由到不同路径，Merge 节点将分支合并。调整条件，观察数据如何分流。

    场景：订单分流处理

    5 条订单进入 IF 节点。金额 ≥ 阈值的走 VIP 路径，其余走普通路径。修改阈值或条件，观察路由变化。

    场景：Merge 合并分支

    两条分支处理完成后，Merge 节点将结果合并为一个数据集继续后续处理。

    场景：错误处理

    n8n 的错误处理策略：

    Continue On Fail：节点出错时，将错误信息附加到数据项中，继续流转到后续节点。适合容错管道。

    Error Trigger：单独的错误工作流，当主工作流出错时自动触发。适合告警和日志记录。

    Retry Settings：节点级别配置重试次数和间隔。适合网络不稳定的外部 API 调用。
